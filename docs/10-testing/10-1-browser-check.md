# 10-1. ブラウザ・デバイス検証 <Badge type="danger" text="必須" />

コーディングの品質を担保するため、対象ブラウザ・デバイスでの表示確認を体系的に行います。確認対象ブラウザの定義は [2章 対応ブラウザ・デバイス](/02-browsers/2-1-required) を参照してください。

## 10-1-1. 検証タイミング

以下のタイミングで必ず確認を行います。途中工程での確認を習慣化することで、手戻りの発生を防ぎます。

| タイミング | 内容 |
|---|---|
| コーディング完了時 | 全ページ・全ブレークポイントでの表示確認 |
| CMS組み込み後 | 実際の入力データでの表示崩れ確認 |
| デザイン変更・機能追加後 | 変更影響範囲の再確認 |

## 10-1-2. 実機・エミュレータの使い分け

### iOS Safari

実機での検証を優先します。実機が確保できない場合は **Xcode の Simulator** を使用します。

🔗 [Xcode Simulatorの導入](https://developer.apple.com/documentation/safari-developer-tools/installing-xcode-and-simulators)

### Android Chrome

実機での検証を優先します。実機が確保できない場合は **Android Studio の Emulator** を使用します。

🔗 [Android Studio](https://developer.android.com/studio?hl=ja)

### Windows ブラウザ

実機またはクラウドサービス（BrowserStack 等）を補助として使用します。

🔗 [BrowserStack](https://www.browserstack.com/)

## 10-1-3. チェック観点

確認時は以下の観点を基準にします。

### 表示・レイアウト

- 余白・フォントサイズ・行間が崩れていない
- 画像が正しいサイズ・位置で表示されている
- 各ブレークポイントでの折り返しが意図通りである
- ホバー・フォーカス・アクティブ状態のスタイルが正しい

### JavaScript

- ブラウザのコンソールにエラーが出ていない
- インタラクション（開閉・スライド・バリデーションなど）が正しく動作する

### フォーム

- 入力・選択・送信が正しく動作する
- バリデーションメッセージが表示される
- 送信後の遷移・表示が正しい

### その他

- 印刷用CSSが指定されている場合は、印刷プレビューでの表示を確認する
- ページのスクロール・アンカーリンクの動作を確認する

## 10-1-4. DoDチェック（納品最低条件）

以下6項目をすべて満たした場合、ブラウザ・デバイス検証の最低限DoDを満たしたものとします。

### A. 動作確認（必須）

- [ ] 必須確認ブラウザ（Windows: Chrome / Firefox / Edge 最新版）で表示崩れ・スクリプトエラーがない
- [ ] モバイル確認対象（プロジェクトで定義した端末・OS）で主要ページの表示崩れがない
- [ ] 主要ブレークポイントでレイアウト崩れ・文言欠け・操作不能がない

### B. 操作確認（必須）

- [ ] 主要インタラクション（開閉・スライド・フォーム送信）が対象環境で動作している
- [ ] ブラウザコンソールにエラーが出ていない
- [ ] 変更対象ページの確認結果（OK/NG）が記録されている

## 10-1-5. 確認環境の記録

確認結果は、対象ブラウザ・OS・端末・確認日・確認者をセットで記録します。特にレガシーブラウザ対応が含まれる案件では、記録を残すことでトラブル対応が容易になります。
